# Spec 034: Randomized Domain `.14` Generator

**Feature:** Library + CLI for generating valid fort.14 mesh bytes from seed + parameters  
**Status:** Draft (Design Phase)  
**MVP Scope:** Rectangle + triangle element type + library-only  
**Effort:** Medium (splittable on element-type axis)  
**Impact:** Unblocks #25 (mesh strategy comparison), #52-53 (test fixtures), #41 (mixed-element validation)

---

## Problem Statement

Current test suite + spec development rely on static meshes:
- Test fixtures hardcoded in `registry_data/manifest.toml`
- Cross-mesh comparison (#25) requires N variants of same domain (manual creation impractical)
- Mixed-element validation (#41) has only 1 test fixture
- Live registry links may rot; example URLs age out

**Solution:** Deterministic generator producing valid fort.14 from seed. Used for:
1. **Test fixtures** — frozen manifest of synthetic meshes
2. **Documentation examples** — reproducible on any machine
3. **Comparative studies** — same domain, tri/quad/mixed variants
4. **Regression validation** — mesh generation stability across releases

---

## Design Principles

1. **Deterministic:** Same seed + params → byte-identical output (across machines/releases)
2. **Barefeet-friendly:** Heavy deps (`numpy`, `triangle`) gated behind `[generate]` extra; base install unchanged
3. **Schema-explicit:** Generated meshes conform to existing fort.14 schema + `VALID_ELEMENT_TYPES` enum
4. **User-friendly:** CLI + library surface both exposed; no internal-only APIs
5. **Extensible:** Easy to add shape primitives (bay, island, channel) + triangulators

## API Surface

### Library

```python
from admesh_domains.generator import random_domain

# Generate from seed
domain, fort14_bytes = random_domain(
    seed=42,
    shape="rectangle",     # "rectangle" | "bay" | "island" | ...
    element_type="triangle",  # "triangle" | "quad" | "mixed"
    node_count=500,
    register=False         # opt-in manifest registration
)

# Use generated bytes
with open("output.14", "wb") as f:
    f.write(fort14_bytes)

# Validate with existing domain/mesh classes
mesh = Mesh.from_fort14_bytes(fort14_bytes)
assert mesh.element_type == "triangle"
```

### CLI

```bash
# Single mesh
admesh-domains generate \
    --output generated.14 \
    --seed 42 \
    --shape rectangle \
    --element-type triangle \
    --node-count 500

# Batch with registration
admesh-domains generate \
    --count 10 \
    --output-dir tests/fixtures/generated/ \
    --seed 42 \
    --shape bay \
    --element-type mixed \
    --register  # Update manifest + commit optional entry

# List available shapes
admesh-domains generate --help-shapes
```

## Implementation Plan (MVP)

### Phase 1: Rectangle Triangles (1-2 hours)
1. **Boundary:** Rectangle polygon [0,1]² corners
2. **Triangulation:** `scipy.spatial.Delaunay` on random interior points
3. **Determinism:** Use NumPy's seeded RandomState
4. **Output:** fort.14 bytes (test with `admesh-domains validate`)

### Phase 2: Quad Support (1 hour)
1. Same rectangle boundary
2. Paving algorithm: subdivide rect → 2×N quad grid + boundary triangles
3. Test: `element_type="quad"` → 0 triangles

### Phase 3: Mixed Elements (1.5 hours)
1. Quad-tree subdivide: some cells stay quad, others split into tris
2. Threshold tuning: ratio of quads vs tris per seed

### Phase 4: Non-rectangular Shapes (2+ hours, Phase 2)
1. **Bay:** Rectangle with semicircle cutout
2. **Island:** Rectangle with polygon island interior
3. **Channel:** Elongated rectangle with width variation
4. Boundary handling per shape

### Phase 5: CLI + Registration (1 hour, Phase 2)
1. Click CLI wrapper
2. Batch generation loop
3. Auto-append to `manifest.toml` (if `--register`)
4. Atomic commit option

## Data Model

### GeneratorParams
```python
@dataclass
class GeneratorParams:
    seed: int
    shape: str  # "rectangle", "bay", "island", "channel", "open-ocean"
    element_type: str  # "triangle", "quadrilateral", "mixed"
    node_count: int  # ~50-10000
    
    def validate(self) -> None:
        assert self.seed >= 0
        assert self.shape in VALID_SHAPES
        assert self.element_type in VALID_ELEMENT_TYPES
        assert 50 <= self.node_count <= 10000
```

### Output
```python
def random_domain(
    seed: int,
    shape: str = "rectangle",
    element_type: str = "triangle",
    node_count: int = 500,
    register: bool = False,
) -> tuple[Domain, bytes]:
    """
    Generate deterministic mesh.
    
    Args:
        seed: Random seed for reproducibility
        shape: Domain boundary shape
        element_type: Triangle, quad, or mixed
        node_count: Approximate node count (exact count depends on algorithm)
        register: If True, add to manifest.toml + commit (requires user intent)
    
    Returns:
        (Domain, fort14_bytes): Domain metadata + binary mesh
    """
```

## Testing Strategy

### Unit Tests
```python
def test_generator_determinism():
    """Same seed → byte-identical output"""
    b1 = random_domain(seed=42, shape="rectangle")[1]
    b2 = random_domain(seed=42, shape="rectangle")[1]
    assert b1 == b2

def test_generator_validates():
    """Output passes admesh-domains validate"""
    domain, b = random_domain(seed=42)
    mesh = Mesh.from_fort14_bytes(b)
    assert mesh.validate() is None

def test_element_type_variants():
    """Each element type generates correct element count"""
    tri_domain, tri_b = random_domain(seed=42, element_type="triangle")
    quad_domain, quad_b = random_domain(seed=42, element_type="quadrilateral")
    # tri has triangles, quad has no triangles
```

### Integration Tests
```python
def test_batch_generation():
    """Batch generate + manifest registration"""
    domains = [
        random_domain(seed=i, element_type="triangle")
        for i in range(5)
    ]
    assert len(domains) == 5
```

## Schema & Registry

### Generated Mesh Entry (manifest.toml)
```toml
[[domains.meshes]]
id = "synthetic-rect-tri@v1-seed-42"
filename = "synthetic_rectangle_triangle_seed42.14"
element_type = "triangle"
kind = "mesh"  # synthetic vs real-world distinction
category = "synthetic"  # domain category
uploaded_date = "2026-05-15"
contributor = "Generator v0.1.0 (seed-42)"
size_mb = 0.02
```

### No Schema Change
- `kind`, `category`, `synthetic` already supported by SCHEMA_VERSION 0.3+
- `contributor` can note generator origin (informational only)

## Out of Scope (Phase 2+)

- Realistic geographic boundary data (coastlines, islands)
- Boundary condition metadata (`.13`, `.15` files; ADCIRC forcing)
- Quality optimization (CVT smoothing, Lloyd iteration)
- Mesh adaptation algorithms
- Auto-registration into live registry (explicit `--register` only)

---

## Risk Assessment

| Risk | Mitigation |
|------|-----------|
| Determinism loss across versions | Unit test with frozen seed; version tag generator output |
| Large node_count → slow generation | Default 500; document scaling; warn on >5000 |
| Invalid fort.14 bytes | Validate with existing `Mesh.from_fort14_bytes()` + schema |
| Dep bloat | `[generate]` extra gates numpy/triangle; base install unaffected |
